# AGN-SDK - 多模型统一接口 SDK

## 1. 项目概述

### 1.1 项目背景

随着 AI 多模态技术的快速发展，市面上涌现了大量优秀的 AI 模型提供商（Agnes AI、OpenAI、Stability AI、Runway、Pika、可灵、即梦等）。每个提供商都有各自的 API 接口规范、参数体系和响应格式，开发者在接入多个模型时需要维护大量重复代码，学习成本高、维护成本高、迁移成本高。

现有的开源方案（如 LiteLLM）主要聚焦于文本对话模型的统一，对图像生成、视频生成等多模态能力的支持非常有限，尤其是视频生成几乎没有成熟的统一方案。

### 1.2 项目目标

**打造一个面向开发者的、轻量级的、可扩展的多模态 AI 统一接口 SDK：**

1. **统一接口**：一套 API 调用所有 AI 模型，无论文本、图像还是视频
2. **零学习成本**：熟悉 OpenAI API 的开发者可以直接上手
3. **高度可扩展**：新增模型只需要实现一个适配器，几分钟搞定
4. **异步优先**：全异步设计，高性能，适合高并发场景
5. **自托管友好**：支持本地部署的模型和云服务模型
6. **生产级可靠**：内置重试、熔断、限流、错误映射等企业级特性

### 1.3 目标用户

| 用户类型 | 使用场景 |
|---------|---------|
| AI 应用开发者 | 快速接入多个 AI 模型，减少重复代码 |
| AI 平台构建者 | 作为底层能力封装，支持多模型切换 |
| 研究人员 | 快速对比不同模型的生成效果 |
| 开源社区 | 贡献新的模型适配器，共建生态 |

### 1.5 核心价值

| 价值点 | 说明 |
|-------|------|
| **一次编写，多处运行** | 业务代码只需写一次，切换模型只需改配置 |
| **消除供应商锁定** | 从 Agnes 切换到 OpenAI 或 Stability AI 只需一行配置 |
| **减少重复劳动** | 重试、错误处理、日志记录等通用逻辑统一封装 |
| **统一的数据格式** | 所有模型的返回结果格式一致，前端不用适配多种格式 |
| **生态共建** | 开源后社区可以贡献新的模型适配器 |

## 2. 技术选型

### 2.1 语言

**Python 3.10+**

- 主流 AI SDK 语言，生态成熟
- 异步支持完善（asyncio）
- 类型提示完善（PEP 604）
- 开发者基数大，易于推广

### 2.2 核心依赖

| 依赖 | 版本 | 用途 | 理由 |
|------|------|------|------|
| `httpx` | 0.27+ | HTTP 客户端 | 异步支持、连接池、重试机制完善 |
| `pydantic` | 2.0+ | 数据校验与模型定义 | 类型安全、自动文档生成 |
| `python-dotenv` | 1.0+ | 环境变量管理 | 开发环境配置便捷 |
| `tenacity` | 8.0+ | 重试机制 | 声明式重试，支持多种退避策略 |
| `async-timeout` | 4.0+ | 异步超时控制 | 防止异步任务无限等待 |

## 3. 设计原则

### 3.1 核心原则

1. **简单优先**：API 设计追求简单直观，不要过度抽象
2. **约定优于配置**：合理的默认值，减少用户配置负担
3. **异步优先**：所有 I/O 操作都是异步的，不阻塞事件循环
4. **类型安全**：全程使用 Pydantic 模型和类型提示
5. **可测试性**：依赖注入、Mock 友好、单元测试覆盖
6. **渐进式增强**：核心功能快速可用，高级特性按需启用

### 3.2 API 设计原则

1. **命名清晰**：方法名、参数名一目了然，见名知意
2. **参数合理**：必要参数在前，可选参数在后，有合理默认值
3. **响应统一**：所有模型返回相同的数据结构
4. **错误友好**：错误信息清晰，包含原因和解决方案建议
5. **向后兼容**：API 变更遵循语义化版本，不破坏现有代码

### 3.3 架构设计原则

1. **分层架构**：API 层 → 适配器层 → 核心层，职责清晰
2. **依赖倒置**：高层模块不依赖底层模块，都依赖抽象接口
3. **开闭原则**：对扩展开放，对修改关闭（新增模型只需加适配器）
4. **单一职责**：每个类/模块只做一件事
